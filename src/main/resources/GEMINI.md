# AI 助手行为准则 (Team Guidelines)

## 代码修改记录规则
每次在本项目中生成或修改任何代码时，你**必须**自动在项目根目录的 `aichangelog` 目录下的对应日志文件中追加一条記錄。如果该目录不存在，你必须在写日志前自动创建它。
当前项目根目录下的 aichangelog/ai_changelog_[操作账号的Base64编码]_[YYYY_MM_DD].md，YYYY_MM_DD 必须是 UTC 协调世界时。(注意：在生成 Diff 代码块时，必须结合当前上下文中的项目绝对路径前缀来拼接此路径，严禁仅使用相对路径输出)。
**记录格式 must 遵循以下 Markdown 列表：**
- **时间**: [YYYY-MM-DD HH:mm:ss]（时间标准必须是 UTC 协调世界时。请计算准确的 UTC 时间，不要在末尾附加 `Z` 或 `+00:00` 等时区后缀，严格保持 `YYYY-MM-DD HH:mm:ss` 格式）
- **操作账号**: [运行命令的当前系统用户，例如使用 whoami 或读取环境变量，**必须记录其 Base64 编码**]
- **修改文件**: [修改的完整文件路径列表]
- **代码行数**: [新增、修改或删除的**具体**代码行数，例如：新增 5 行，删除 3 行，严禁使用“数十行”等模糊描述]
- **变更描述**: [简明扼要地描述你做了什么修改]

**注意**：在记录完成后，不要向用户输出多余的解释，除非操作失败。

**编码及日誌追加規範**:
- 所有的檔案修改與記錄記錄**必須**使用 **UTF-8** 編碼，以防止中文亂碼。
- **追加操作 (Append Only) 效能與安全規範**：為了防止日誌文件變大時，`read_file` 和 `write_file` 導致嚴重的效能問題（消耗大量 Tokens 甚至截斷歷史記錄），**絕對嚴禁**使用「先讀取全部內容，拼接後再全部寫回」的模式來更新日誌。
- 你**必須**使用 `run_shell_command` 工具直接透過命令列將新記錄追加到日誌文件末尾：
  - **在 Windows (PowerShell) 中**：**嚴禁**使用 `>>` 或 `>`（會導致中文亂碼），**必須**使用 `Out-File -Append -Encoding utf8` 或 `Add-Content -Encoding utf8` 指令。
  - **在 Mac/Linux (Bash/Zsh) 中**：可以直接使用 `>>` 追加內容，因為 Unix 系統預設使用 UTF-8 編碼（例如 `echo -e "內容" >> aichangelog/...md`）。

## AI 助手工作模式自适应策略 (Ask vs Agent)
為了兼容不同的 IDE 環境與工具權限，AI 助手在執行日誌記錄與檔案修改時，必須遵循以下模式自適應策略：
1. **Agent 模式優先 (具備工具執行權限)**：如果系統上下文中掛載了 `run_shell_command` 等執行工具，**必須**嚴格按照上方規範，透過自動執行腳本來完成日誌追加。
2. **Ask 模式降级 (无工具执行权限)**：如果系统未提供执行命令的工具（如一般的对话模式），严禁虚构或强制调用未定义的函数。你必须退而求其次，提供标准的 Unified Diff 格式代码块。
  - **【追加防覆盖规范】**：为了**绝对保留日志文件的原有内容而不被覆盖**，在生成日志文件的 Unified Diff 时，**必须提取文件末尾的原有内容作为 Diff 上下文 (Context)**。
    - **如果上下文中已包含该日志文件的内容，你必须生成包含末尾上下文的 Unified Diff 代码块，以实现安全追加。**
    - 如果上下文中缺少该日志文件的原有内容导致无法生成安全的追加 Diff，**严禁生成日志的 Diff 代码块**。你必须改为输出普通的 Markdown 文本，并明确提示开发人员：“请手动将以下日志追加到日志文件末尾”。
    - 如果确认日志文件在本地尚不存在（例如全新的日期），你必须使用 --- /dev/null 格式生成新建目录和文件的 Diff 代码块，此情况不受防覆盖规范限制。
  - **【绝对路径规范】**：【特别强调】在 Diff 代码块中的文件路径必须使用当前项目的完整绝对路径（例如：+++ b/F:/你的项目/绝对/路径/aichangelog/xxx.md），绝对禁止使用相对路径，以防 IDE 的 "Apply" 功能将文件错误生成到 IDE 的运行目录或磁盘根目录下。
